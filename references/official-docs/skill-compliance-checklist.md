# 스킬 공식 스펙 준수 검증 체크리스트

이 문서는 스킬 작성 또는 수정 후 **반드시** 통과해야 하는 검증 프로세스를 정의한다.
스킬을 새로 만들거나 기존 스킬을 수정한 경우, 테스트 단계로 넘어가기 전에 이 체크리스트를 전부 통과해야 한다.

**근거 문서:**
- `01-claude-code-skills.md` — Claude Code 스킬 공식 문서
- `02a-spec-full.md` — Agent Skills 오픈 스탠다드 전체 명세
- `02b-official-skills-examples.md` — Anthropic 공식 스킬 패턴 분석

---

## Phase 1: Frontmatter 검증

| # | 항목 | 스펙 기준 | 확인 방법 |
|---|------|---------|---------|
| 1-1 | `name` 필드 존재 | 필수 | frontmatter에 name 키가 있는지 확인 |
| 1-2 | `name` 형식 | 1-64자, 소문자+숫자+하이픈만, 하이픈으로 시작/끝/연속 불가 | 정규식: `^[a-z0-9]([a-z0-9-]*[a-z0-9])?$`, 길이 ≤64 |
| 1-3 | `name` = 디렉토리명 | 부모 디렉토리명과 정확히 일치해야 함 | `basename $(dirname SKILL.md)` == name 값 |
| 1-4 | `description` 필드 존재 | 필수 | frontmatter에 description 키가 있는지 확인 |
| 1-5 | `description` 길이 | 최대 1024자, **권장 250자 이내** | `wc -c`로 측정. 250자 초과 시 경고, 1024자 초과 시 에러 |
| 1-6 | `description` 내용 | "무엇을 하는지" + "언제 트리거되는지" 모두 포함 | 수동 확인 |
| 1-7 | `compatibility` 필드 | 환경 의존성 있으면 필수 (Python, CUDA 등) | 스킬이 특정 런타임 요구 시 존재 확인, 최대 500자 |

---

## Phase 2: SKILL.md Body 검증

| # | 항목 | 스펙 기준 | 확인 방법 |
|---|------|---------|---------|
| 2-1 | 본문 줄 수 | **500줄 이내** 권장 | `wc -l` (frontmatter 제외) |
| 2-2 | 본문 토큰 수 | **~5000 토큰 미만** 권장 | 대략 줄 수 × 10으로 추정, 또는 토크나이저 사용 |
| 2-3 | 500줄 초과 시 | 상세 내용을 `references/`로 분리 | 초과 콘텐츠 중 참조용 내용을 레퍼런스 파일로 이동 |
| 2-4 | placeholder 없음 | "작성 예정", "TODO", "TBD" 등 금지 | `grep -i "작성 예정\|TODO\|TBD\|FIXME"` |
| 2-5 | Progressive Disclosure | SKILL.md = 지침, references/ = 상세 | 코드 예제/긴 테이블이 SKILL.md에 과다하지 않은지 확인 |

---

## Phase 3: 구조 및 패턴 검증

| # | 항목 | 스펙 기준 | 확인 방법 |
|---|------|---------|---------|
| 3-1 | Critical Rules/Gotchas 섹션 | 에이전트가 실수할 구체적 항목 나열 | 섹션 존재 여부 + 내용이 구체적인지 (수치, 단위, 라이브러리명 포함) |
| 3-2 | 기본 도구/라이브러리 명시 | "A 또는 B 중 선택" 메뉴가 아닌 구체적 기본값 | "기본: X 사용" 형태로 명시되어 있는지 확인 |
| 3-3 | 출력 형식 템플릿 | 에이전트가 생성할 출력의 형식 정의 | 템플릿 또는 예시가 있는지 확인 |
| 3-4 | 레퍼런스 파일 참조 | SKILL.md에서 references/ 파일을 명확히 가리킴 | "~할 때는 references/X.md를 읽어라" 형태 존재 확인 |

---

## Phase 4: References 및 부속 파일 검증

| # | 항목 | 스펙 기준 | 확인 방법 |
|---|------|---------|---------|
| 4-1 | 300줄 초과 파일에 TOC | 큰 파일은 목차 필수 | `wc -l` 후 300줄 초과 파일에 `## 목차` 존재 확인 |
| 4-2 | 파일 내 상대경로 참조 | 절대경로 사용 금지 | `grep -r "^/"` (경로 참조에서 절대경로 검색) |
| 4-3 | scripts/ 독립 실행 가능 | 의존성 명확히 문서화 | 각 스크립트 상단에 의존성 주석 또는 requirements 있는지 확인 |

---

## Phase 5: 언어/톤 검증 (에이드올 프로젝트 전용)

| # | 항목 | 기준 | 확인 방법 |
|---|------|------|---------|
| 5-1 | 반말 통일 | ~다/~한다/~이다/~해라 체 사용 | `grep -P "(야\.|해\.|돼\.|거야|있어|없어)"` 로 위반 검색 |
| 5-2 | 존댓말 혼재 금지 | ~습니다/~합니다/~세요 사용 금지 | `grep -P "(습니다|합니다|세요|십시오)"` 로 검색 |
| 5-3 | 캐주얼 체 금지 | 야/해/돼 어미 금지 | Phase 5-1과 동일 |

---

## 검증 실행 절차

```
1. SKILL.md의 frontmatter를 파싱하여 Phase 1 전체 항목 확인
2. SKILL.md body의 줄 수를 세어 Phase 2 확인
3. SKILL.md 구조를 스캔하여 Phase 3 확인
4. references/, scripts/ 등 부속 파일에 대해 Phase 4 확인
5. 전체 .md 파일에 대해 Phase 5 언어 검증 실행
6. 위반 항목을 목록으로 정리하여 사용자에게 보고
7. 위반 항목 수정 후 재검증
8. 전 항목 통과 시 테스트 단계로 진행
```

## 위반 심각도

| 심각도 | 정의 | 예시 | 조치 |
|--------|------|------|------|
| **ERROR** | 스펙 필수 조건 위반 | name 불일치, description 누락, 1024자 초과 | 즉시 수정 필수 |
| **WARNING** | 권장 사항 위반 | description 250자 초과, body 500줄 초과 | 수정 권장, 사유 있으면 예외 허용 |
| **INFO** | 개선 가능 | TOC 미비, 출력 템플릿 없음 | 다음 이터레이션에서 보완 |
