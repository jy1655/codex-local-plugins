# Oracle Consult

## Overview

복잡한 기술 결정, 반복 실패, 보안/성능 우려에 대해 딥 리즈닝 기반 전략적 자문을 수행한다.
구현하지 않는다. 분석하고 추천한다.

## When to Use

- 아키텍처 결정에서 여러 시스템 간 트레이드오프가 있을 때
- 수정 시도가 2회 이상 실패한 후
- 보안 또는 성능 우려가 있을 때
- 익숙하지 않은 코드 패턴을 만났을 때
- 대규모 구현 완료 후 자체 리뷰가 필요할 때

사용하지 않을 때:
- 단순 파일 작업 (직접 도구 사용)
- 첫 시도 전 (먼저 직접 시도)
- 코드에서 답을 읽을 수 있는 질문
- 사소한 결정 (변수명, 포매팅)

## Quick Reference

| 상황 | 자문 초점 |
|------|----------|
| 아키텍처 결정 | 트레이드오프 분석, 장단기 영향, 추천 경로 |
| 반복 수정 실패 | 근본 원인 추적, 놓친 가정 식별, 대안 접근 |
| 보안/성능 우려 | 위험 표면 분석, 완화 전략, 우선순위 |
| 자체 리뷰 | 설계 결함, 엣지 케이스, 누락된 고려사항 |

## Consultation Protocol

### 1. 문맥 정리

자문 요청 시 포함할 것:
- 현재 상황과 시도한 접근
- 관련 코드 (diff 또는 파일 전문)
- 제약 조건 (기술 스택, 호환성, 일정)
- 구체적 질문 또는 결정 포인트

### 2. 분석 구조

```
문제 정의: 무엇을 결정/해결해야 하는가
현재 상태: 지금까지 시도한 것과 결과
선택지: 가능한 접근들 (각각의 장단점)
추천: 추천 경로와 이유
위험: 추천 경로의 알려진 위험과 완화 방법
다음 단계: 구체적 실행 항목
```

### 3. 자문 결과 활용

- 추천은 구체적이고 실행 가능해야 한다
- "상황에 따라 다르다"로 끝나지 않는다
- 추천에 동의하지 않을 수 있는 이유도 명시한다

## Common Mistakes

- 자문이 아니라 구현을 시작하는 것 — 분석만, 코드 수정 금지
- 첫 시도도 안 해보고 자문부터 요청하는 것 — 먼저 직접 시도
- 문맥 없이 추상적 질문만 던지는 것 — 코드와 제약 포함
- "정답은 없다"로 끝내는 것 — 반드시 추천과 이유를 제시
- 코드에서 바로 읽을 수 있는 것을 질문하는 것 — 직접 읽기
