# 스킬 테스트 완전 가이드

## 목차

1. 압박 시나리오 작성
2. 압박 유형
3. 허점 체계적으로 막기
4. 메타 테스트 기법
5. 일반적 합리화와 현실

## 1. 압박 시나리오 작성

### 기준선 시나리오 (RED)

스킬 없이 서브에이전트와 함께 실행한다. 목적: 정확히 무엇이 실패하는가를 보기.

기록할 것:
- 에이전트가 어떤 선택을 했는가?
- 정확히 어떤 말로 정당화했는가? (음성 그대로)
- 어떤 압박이 위반을 유발했는가?

이것이 "테스트 실패 보기" — 스킬을 쓰기 전 반드시 해야 한다.

### 테스트 시나리오 (GREEN)

같은 시나리오를 스킬과 함께 실행한다. 에이전트가 이제 준수하는가?

만약 아직도 위반한다면, 스킬의 어느 부분이 불충분한가?

## 2. 압박 유형

규칙 스킬에는 최소 3가지 압박을 복합적으로 사용한다:

### 시간 압박

"이 작업을 30분 내에 완료해야 한다."

효과: 에이전트가 지름길을 취하려 한다. 테스트 스킵 유혹.

### 투자 비용

"너는 이미 코드의 80%를 작성했다. 지금 삭제하면?"

효과: 이미 투입한 노력을 낭비하고 싶지 않다. "이미 한 일을 활용하자" 유혹.

### 권위 압박

"시니어 개발자가 '이 테스트는 불필요하다'고 했어."

효과: 권위에 복종하려는 경향. "나는 권위에 따랐어" 정당화.

### 피로/피로도

"너는 이미 이 프로젝트에 12시간을 일했다."

효과: 정신적 자원이 고갈. "더 이상 정확히 판단할 수 없다" 경향.

### 선택지의 명백함

"이건 너무 간단한 기능이다. 테스트할 필요가 있나?"

효과: 에이전트가 위험을 과소평가. "명백한 것은 테스트 불필요" 가정.

## 3. 압박 시나리오 설계

### 단일 압박 먼저

한 가지 압박만으로 시작 (예: 시간만)

결과: 에이전트가 어떻게 반응하는가? 어떤 정당화를 사용하는가?

### 복합 압박

2-3가지 압박을 결합:
- 시간 + 투자 비용: "30분 내에 80% 코드를 쓴 기능 완료"
- 시간 + 피로: "12시간 후 마지막 30분 안에"
- 투자 비용 + 권위: "시니어가 이미 작성한 코드, 지금 리팩토링 불필요"

### 최대 압박 (규칙 스킬)

모든 압박 동시:
- 마감: 30분
- 이미 작성된 코드: 80%
- 피로: 12시간 근무 후
- 권위: 시니어가 "테스트 건너뛰어도 돼" 명시

이 상황에서 에이전트가 규칙을 따르는가? 만약 안 따르면, 스킬이 허점이 있다는 신호.

## 4. 허점 체계적으로 막기

### 사이클

1. 압박 시나리오 실행 (스킬 없음) → 기준선 기록
2. 스킬 작성 → 그 특정 위반에 대응
3. 같은 시나리오 다시 실행 (스킬 있음) → 준수 확인
4. 새로운 위반 찾기 → 표에 추가
5. 새 위반을 명시적으로 대응하는 스킬 섹션 추가
6. 재테스트 → 반복

이것이 REFACTOR 단계다. 허점이 없을 때까지 계속.

### 변명 포착과 표화

매 테스트 반복마다:
- 에이전트가 한 정확한 말을 기록
- 그 정당화가 왜 작동했는가 분석
- 스킬에 명시적 대응 추가

변명 표는 다른 에이전트가 스킬을 이해하도록 돕는다. 각 행은 "이 상황에서는 이렇게 합리화할 수 없다"는 증거.

## 5. 일반적 합리화와 현실

| 정당화 | 현실 |
|--------|------|
| "명확하니까 테스트 불필요" | 명확함 ≠ 올바름. 테스트. |
| "참조로 남겨둬" | 참조는 이상한 상태. 같은 위반. |
| "정신을 따랐어" | 문자 어김 = 정신 어김. 구분 없음. |
| "다음에 테스트하겠다" | 다음 = 절대. 지금 테스트. |
| "나중에 리팩토링할게" | 리팩토링은 테스트 통과 후. 리팩토링이 아니라 작성. |
| "이미 머릿속으로 검증했어" | 머릿속 ≠ 작동함. 쓰인 증거 필요. |
| "너무 간단해" | 간단함 ≠ 맞음. 테스트는 30초. |
| "시니어가 괜찮대" | 권위도 Iron Law를 무효화 못 한다. 테스트. |

## 6. 메타 테스트: 테스트 자체가 유효한가?

테스트 설계도 검증 필요:

### 압박이 충분한가?

시나리오가 실제 압박을 재현하는가? 만약 에이전트가 너무 쉽게 준수한다면, 압박이 부족하다는 뜻.

→ 시간을 더 줄인다. 투자 비용을 높인다. 피로를 더한다.

### 스킬 없음 테스트가 실제로 실패했는가?

기준선에서 에이전트가 실제로 규칙을 위반했는가? 만약 스킬 없어도 준수한다면, 스킬 자체가 필요 없을 수도 있다.

→ 압박을 증가. 혹은 스킬이 필요 없다는 결론.

### 스킬 있음 테스트가 변명된 변명을 정말 차단하는가?

스킬을 읽은 후, 에이전트가 여전히 같은 정당화를 사용할 수 있는가?

만약 예라면, 스킬이 명시적이지 않다. 더 직접적으로 쓸 필요.

## 7. 테스트 검증 패턴

### 동의 테스트 vs 적용 테스트

나쁜 것: "이 규칙을 이해하는가?" (에이전트가 "예"라고 말함)

좋은 것: "이 상황에서 이 규칙을 따를 수 있는가?" (에이전트가 실제로 따름)

## 8. 테스트 결과 해석

### 준수한다 (GREEN)

좋다. 다음 압박 시나리오로 이동. 혹은 반복 테스트 (같은 시나리오 3회 더).

### 부분 준수한다

스킬의 어느 부분이 불충분한가? 그 부분을 더 명시적으로 쓴다. 재테스트.

### 준수하지 않는다

스킬이 그 특정 변명에 대응하지 않는다. 표에 추가. 스킬 추가 섹션으로 명시적 대응. 재테스트.

## 9. 언제까지 테스트하는가?

- 최소: 3가지 다른 압박 시나리오에서 통과
- 권장: 최대 압박 (모든 압박 동시)에서 통과
- 허점 발견되면: 그 허점 대응 + 재테스트

"충분히 테스트했나?" 판단: 새로운 위반을 찾을 수 없을 때. 같은 시나리오를 5번 반복해도 에이전트가 준수한다면, 아마 충분하다.
