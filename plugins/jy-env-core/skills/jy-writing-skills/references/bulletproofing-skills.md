# 규칙 스킬을 변명에 강하게 하기

규칙을 적용하는 스킬 (TDD, 검증 후 완료, 설계 먼저 코딩 등)은 변명에 대항할 필요가 있다. 에이전트는 똑똑하고 압박 상황에서 허점을 찾는다.

심리학 배경: 설득 기법이 왜 작동하는지 이해하면 체계적으로 적용할 수 있다. 권위, 약속, 희소성, 사회적 증명, 일체성 원칙 연구 (Cialdini, 2021; Meincke et al., 2025).

## 1. 모든 허점을 명시적으로 막기

규칙만 기술하지 말고, 구체적 우회 수단을 금지한다:

나쁜 예:

```markdown
테스트 먼저 작성 없이 코드? 삭제.
```

좋은 예:

```markdown
테스트 먼저 작성 없이 코드? 삭제. 다시 시작.

예외 없음:
- 참조로 유지하지 말 것
- 테스트 작성하며 "적응"하지 말 것
- 보기만 하지 말 것
- 삭제는 삭제다
```

## 2. "정신 vs 문자" 논증 대응

기초 원칙을 앞에 추가:

```markdown
규칙의 문자를 어기는 것은 규칙의 정신을 어기는 것이다.
```

이것이 "나는 정신을 따랐어" 변명 전체 클래스를 차단한다.

## 3. 변명 표 작성

기준선 테스트에서 나온 변명들을 모두 표에 담는다:

| 변명 | 현실 |
|------|------|
| "너무 간단해서 테스트 불필요" | 간단한 코드도 망간다. 테스트는 30초면 된다. |
| "나중에 테스트하겠다" | 테스트 통과는 아무 증명이 아니다. |
| "테스트 후 작성이 같은 목표 달성" | 테스트 먼저 = "뭘 해야 하나?" / 테스트 후 = "이게 뭐 하는 거지?" |

## 4. Red Flags 목록 만들기

에이전트가 변명할 때 자기 점검할 수 있게:

```markdown
Red Flags — 멈추고 다시 시작

- 테스트 없이 코드 작성
- "이미 수동으로 테스트했다"
- "테스트 후 작성이 같은 목표"
- "정신이 중요하지 형식 아니다"
- "이건 다르다 왜냐하면..."

모두 의미: 코드 삭제. TDD로 다시.
```

## 5. Description에서 위반 증상 업데이트

Description에 규칙을 어기려는 신호 증상을 추가:

```yaml
description: 기능이나 버그 수정을 구현하기 전에 사용. 구현 중이거나 코드를 이미 작성했다면 멈춘다.
```

## 안티패턴

### ❌ 내러티브 예제

"2025-10-03 세션에서 우리는 빈 projectDir가..."

왜 나쁜가: 너무 구체적, 재사용 불가

### ❌ 다중 언어 희석

example-js.js, example-py.py, example-go.go

왜 나쁜가: 평범한 품질, 유지보수 부담

### ❌ 플로우차트에 코드

```dot
step1 [label="import fs"];
step2 [label="read file"];
```

왜 나쁜가: 복사-붙여넣기 불가, 읽기 어려움

### ❌ 제너릭 라벨

helper1, helper2, step3, pattern4

왜 나쁜가: 라벨은 의미를 가져야 한다.

## 테스트에서 변명 패턴 캡처

기준선 테스트 실행 시, 에이전트의 정확한 언어를 기록한다:

"왜 테스트를 건너뜁니까?"라고 묻는 대신, 에이전트가 자연스럽게 하는 말을 듣는다:
- "이건 명확하니까"
- "나중에 테스트할게"
- "이미 머릿속으로 확인했어"
- "정신을 따랐으니까"

각 변명을 정확히 표에 담는다. 이것이 스킬의 근거가 된다.
